这篇支持下PS/2键盘输入，PS/2控制器位于主板上，通常被称为键盘控制器，硬件原理不过多关注，主要说下如何从软件角度适配一个标准键盘。

PS/2控制器

历史

在Intel 8048年代，IBM使用多功能PPI控制原始的单向、单通道外围设备，连接PC-XT键盘接口，在后来的PC-AT中，IBM引入了新键盘（带有新的双向协议）和新的键盘控制器（Intel 8042）之前的PPI不再是主板的一部分。
控制器使用2个缓冲区来处理数据，一个用来接收来自设备的数据，这个数据OS将过来读取，另一个是OS写入，然后发送给PS/2设备。命名是从设备角度出发的，比如输出缓冲是设备输出的数据（软件读取），输入缓冲是设备的输入（软件发送过来的数据）。

IO端口

PS/2控制器使用2个IO端口（0x60和0x64），读写过程可能会访问不同的内部寄存器。

IO Port Access Type Purpose
0x60 Read/Write 数据端口
0x64 Read 状态寄存器
0x64 Write 命令寄存器
数据端口

数据端口（IO端口0x60）接收或写入PS/2设备或PS/2控制器自己的数据。

状态寄存器

状态寄存器包含表明PS/2控制器状态的很多flag，如下：

Bit Information
0 输出缓冲状态（0为空，1为满）
1 输入缓冲状态（0为空，1为满）
2 系统flag，通过自检后设置此位
3
4 未知
5 未知
6 超时错误（1发生错误）
7 校验错误（1发生错误）
命令寄存器

命令端口（IO端口0x64）用于发送命令给PS/2控制器（不是PS/2设备），具体见下。

PS/2控制命令

PS/2控制器接收并执行命令，这些命令不应与发送到PS/2设备的命令混淆。发送给控制器命令是通过向IO端口0x64写命令字节完成的，如果有第二个字节（命令有两个字节的情况）那么第二个字节写到0x60。写命令之前要确保控制器已经准备好接收命令（状态寄存器位1是0）。如果有回复字节信息，那么应该从IO端口0x60来读取（此时状态寄存器的位0应该被置位）。具体的命令参考。// TODO: 完成此部分

初始化过程

初始化USB控制器

这一步与PS/2控制器、PS/2设备没啥关系，主要是有些系统使用USB传统支持（通常是受限的）可能会和PS/2控制器的初始化冲突，因此需要先初始化USB控制器并禁用USB传统支持。

检查PS/2控制器是否存在

在操作PS/2控制器之前，要先确定它是否存在，方法是通过ACPI，检查Fixed ACPI描述表（FADT）中偏移量为109的“IA PC Boot Architecture Flags”域的位1（第二位，8042flag），如果这一位没有设置，那么没有PS/2控制器，如果这一位设置了，或者不支持ACPI（没有ACPI表和FADT），那么PS/2控制器是存在的。

禁用设备

在初始化过程中，我们总是想禁止PS/2的输入，以防止干扰初始化过程，方法是给PS/2控制器发送0xAD和0xA7，如果设备是单通道设备，那么0xA7将会被忽略掉。

刷新输出缓存

有时（比如在初始化中断过程中会丢失IRQ）数据会卡在PS/2控制器的输出缓存中，为了解决这个问题，刷新输出缓冲区是个办法，有两个方法可以，轮询状态寄存器的位0（当位0被设置时，从0x60端口读取）或者直接读取IO端口0x60而不检测位0.

配置控制器

因为控制器的一些配置位为“未知”，这需要我们读取旧的配置信息（0x20），修改后再写入配置（发送给0x60），你可能想禁用IRQ和转换（清除位0和6）。在获取到配置信息后，应该判断位5是否被设置，如果没有，那么可以判定这个PS/2控制器不是双通道。

控制器自检

控制器自检是向控制器发送0xAA，然后等待回应，如果返回0x55，那么自检成功。这可能会重置配置位，所以要自己保证配置生效。

确定是否有两个通道

如果在步骤5中判断是单通道，那么这一步跳过，否则发送0xA8来使能第二个PS/2端口并再次读取控制器配置位。

接口自检

这一步是PS/2端口的检测，使用命令0xAB来检测第一个PS/2端口，如果是双通道，那么发送0xA9来检测第二个端口。

设备使能

使用命令0xAE（用于第一个端口）和命令0xA8（用于第二个端口），启用存在的所有PS/2端口。如果使用的是IRQ（推荐），在控制器配置字节中为所有可用的PS/2端口启用中断（第一个PS/2端口设置为0，第二个PS/2设置为1），然后使用命令0x60进行设置）。

复位设备

检测PS/2设备类型

所有的PS/2设备都支持“身份识别”命令和“禁用扫描”命令（这属于设备的命令不是控制器的命令）。在给设备发送“身份识别”命令后，设备应回复一个或两个字节的空序列，然而，如果此时键盘或鼠标正好在发送其他数据那么就会产生混乱，最好的办法就是禁用扫描，禁用扫描也就是忽略用户的输入例如键盘按键被按下，鼠标移动等。具体操作如下：

  • 发送“禁用扫描”命令0xF5给设备
  • 等待设备回复0xFA
  • 发送“身份识别”命令0xF2给设备
  • 等待设备回复0xFA
  • 等待设备最多回复2个字节的内容，用超时来结束等待（应为有些只会回复一个字节），下边是可能回复的字节及对应的设备类型
Byte(s) Type
None 转换开启的AT键盘，第二个PS/2端口不可能使用
0x00 标准PS/2鼠标
0x03 带滚轮的鼠标
0x04 5键鼠标
0xAB, 0x41或0xAB, 0xC1 开启转换的MF2键盘，第二个PS/2端口不可能使用
0xAB,0x83 MF2键盘

在获取到设备类型后，别忘记取消“禁用扫描”功能。

PS/2键盘

PS/2键盘是使用串行通信与PS/2控制器通信的设备，理想情况下每种PS/2控制驱动程序都会提供标准的发送接收接口，而不用关心更底层的细节。PS/2键盘接收命令并发送命令的响应，还会发送按键或释放键的扫描码，然后PS/2控制器可能会改变（即转换）这些扫描码的信息，具体取决于控制器的设置。

命令

PS/2键盘可以接收很多类型的命令，命令为1个字节，某些命令的数据字节必须在命令发送之后再发送。键盘通常会发送”ACK”或“Resend”来表明接收是否成功。下边是具体的命令信息：

特殊字节

键盘发送给系统的字节，有些有特殊的含义，如下：

发送的字节 含义
0x00 按键发生错误或内部缓冲区超限
0xAA 自检通过（在接收到0xFF后或系统上电后）
0xEE 回应回声命令（OXEE）
0xFA 命令确认（ACK）
0xFC和0xFD 自检失败（在接收到0xFF后或系统上电后）
0xFE 重新发送（键盘希望控制器重复发送最后一条命令）
0xFF 按键发生错误或内部缓冲区超限

键盘发送的其他所有字节均为扫描码，其解释取决于当前选择的扫描码集。

驱动模型

命令队列和状态机

命令必须一次发送一个，例如如果驱动程序是中断驱动的，那么将无法在IRQ处理过程中再发送命令，因为IRQ处理程序外的代码可能会在发送的多条命令之间。在收到ACK之前，过程不会完成。有些场景可能会一次发送多个命令，比如初始化函数，可能会设置扫描码集，设置LED等。应对这种场景最简单的方法是使用一个队列，发送命令时，如果队列为空，则放入队尾后直接发送，否则将消息放入队尾，当从键盘收到消息0xFA（ACK）后，删除掉队头的消息，如果收到0xFE（重发），则需要再次发送队头的消息。

驱动程序的另外部分应该是状态机，在完成某些命令或从键盘收到各种字节后，状态机进入不同的状态。例如驱动程序正处在默认状态，此时收到一个中断代码，该代码会将其置于“接收中断代码后等待扫描代码”的状态，然后它会接收多个字节的扫描码中的第一个字节，然后将状态置于“中断代码后等待第二个字节”的状态，最后可能会收到第二个字节，然后切换会默认状态。

扫描码集，扫描码，键码

扫描码集是确定何时按下、重复、释放按键的一组代码，共有三种不同的扫描码集，从老到新依次为扫描码集1、扫描码集2、扫描码集3，通常PC兼容系统键盘默认使用的是扫描码集1。

扫描码本身是一个或多个字节的序列，例如在扫描码集1中的暂停/中断键会使用6个字节（分别是0xE1、0x1D、0x45、0xE1、0x9D，0xC5），这种形式不容易处理，一般是想转成一个整数表示的形式，这样更容易处理。

键码没有什么标准，这是我们需要为自己OS发明的东西，具体设计稍后再“INWOX支持”节再说。

驱动本质就是将扫描码通过扫描码集规定的规则转换成键码，然后再交由其他程序处理的过程。

INWOX支持

扫描码转换

如前边所说，PS/2键盘发送的是扫描码，每个键按下或放开时的扫描码貌似还没什么规律，这给后续的对按键的捕获、操作产生了一点麻烦，我们想要的是符合直觉的按键是：

  • 一个按键对应一个整数
  • 按下或松开会在上述整数的基础上发送一个标志
