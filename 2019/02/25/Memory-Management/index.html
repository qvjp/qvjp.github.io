递归页目录

访问内存是基于分页机制的，访问的是虚拟内存，通过页目录和页表将其转换成物理地址，但页目录和页表在虚拟内存中该如何存放呢？放到哪里合适呢？当然，可以保留一部分虚拟空间，然后将页表映射到这里，当每次创建一个新页表，就需要把它的物理地址写到页目录中，同时还要给它映射一个虚拟地址，当这个页表不再需要，还需要思考如何取消这个映射，这样没问题，但可能逻辑上会很复杂，所以递归映射页目录应运而生。
递归页目录就是将页目录映射到自己，听起来很绕，其实不难。在初始化时，将页目录中最后一项指向页目录本身的物理地址，如下汇编代码：

这里很有意思，因为页目录和页表都是1024个32位大小内存，且结构基本一致，所以在处理器看来，页目录中最后一项所指就是一个页表（其实是页目录本身）。每个页表可以覆盖4MB的内存，最后一个页目录项指向的页表（页目录本身）正好覆盖最后的4MB内存，也就是0xFFC0 0000到0xFFFF FFFF。因为现在用页目录作为了一个页表，那访问0xFFC0 0000会怎样呢？首先，0xFFC0 0000对应二进制的高10位是1111 1111 11，也就是对应的页目录项是最后一个，然后这个页目录项指向的页表就是页目录本身（0xFFFF F000），0xFFC0 0000对应二进制的中间10位是0000 0000 00，也就是要找页表的第0项，高10位所指向的页表（就是页目录）的第0项指向的地址就是真正页表0的物理地址，也就是0xFFC0 0000！所以，现在可以访问虚拟地址空间全部的页表。
所以，现在将页表映射到虚拟地址空间的高4MB（0xFFC0 0000开始），本来页表和页目录一共会占用4MB+4KB的空间，巧的是地址空间的最高4KB被映射到页目录，这样只需4MB就可以了，然后通过虚拟地址0xFFFF F000加偏移量就可以访问页目录。下图可能会让你对这块知识更加清楚：

物理内存管理

终于到了物理内存管理，说了这么多，就是为了管理物理内存，好了，开始。
物理内存管理器要做的是当你需要时可以给你返回一个新页框，调用者通常不会关心具体返回那个页，只要返回可用的页就可以。所以要实现物理内存管理首先要考虑的就是如何管理这些页，管理方法有很多，都不是我发明的（😂，但我想多介绍几个，最后INLOW将采用比较简单的栈来管理物理页，后期可能会更改管理策略，但不是现在。

Bitmap

这是最直接简单的管理策略，具体就是使用一块N/8字节的空间作为位图（N单位大小内存的数目），然后通过每一位的0/1来判断该位对应的内存块是否被使用。设置一个给定页的状态很快（O(1)），但是要分配一个新的页就需要遍历位图，时间复杂度也就变成了O(N)。

Stack-based memory allocation

用栈来管理内存就是把每一个可用的物理块地址放到栈结构中，分配页时使用pop操作，回收页时使用push操作，这样分配或释放内存都是O(1)，因为都是在栈顶操作。但这种方式不能检查页是否被分配，除非再借助位图来存放状态。
